# Архитектурный обзор — высокоуровневая схема системы VK Identity Checker

## 1. Назначение документа
Этот документ описывает высокоуровневую архитектуру системы идентификации профилей ВКонтакте: компоненты, потоки данных, границы модулей и взаимодействия.

## 3. Компоненты системы

### 3.1. VK API Client (`src/vk_api/vk_api_client.py`)
- **Ответственность**: взаимодействие с API ВКонтакте
- **Функции**: `get_profile()`, `get_friends()`, `get_photos()`
- **Зависимости**: `requests`
- **Граница**: изолирован от логики сравнения, предоставляет сырые данные словарями

### 3.2. Слой Утилит (`src/utils/`)
- `config.py` — настройки (пути, пороги, API-токены)
- `logger.py` — структурированное логирование
- `utils.py` — вспомогательные функции (кэширование, валидация)
- `data_preparer.py` — подготовка данных перед подачей в матчеры

### 3.3. Слой Обработки / Handlers (`src/handlers/`)
- `friends_handler.py` — получение и нормализация списка друзей
- `photo_handler.py` — получение и нормализация фотографий
- `file_exporters.py` — экспорт в JSON/HTML

### 3.4. Слой Matchers (`src/matchers/`)
7 независимых модулей со стандартным интерфейсом:
- `name_matcher.py` — сравнение имён/фамилий
- `geo_matcher.py` — сравнение локаций
- `friends_matcher.py` — Jaccard по друзьям
- `social_geo_analyzer.py` — центроиды + плотность
- `visual_matcher.py` — face recognition + метаданные
- `content_matcher.py` — интересы + стилистика
- `demographics_matcher.py` — дата рождения, пол и т.д.

**Контракт матчера**:
```python
class BaseMatcher:
    def compare(self, data1, data2) -> Dict[str, Any]:
        # Обязательные поля в результате:
        #   'score': float [0, 1]
        #   'has_data': bool
        #   'interpretation': str
```

### 3.5. Оркестратор (`src/matchers/profile_comparer.py`)
- **Вход**: 2 профиля + необязательные данные (друзья, фото)
- **Логика**:
  1. Инициализирует 7 матчеров (Dependency Injection)
  2. Вызывает каждый `_analyze_*()` метод
  3. Собирает взвешенную сумму (веса по умолчанию: 15/25/20/8/7/15/10)
  4. Вычисляет confidence на основе `has_data`
  5. Формирует интерпретацию
- **Выход**: структурированный dict с `final`, `analysis`, `scores`, `breakdown`

### 3.6. Слой Вывода (`src/output/`)
- `html_generator.py` — рендеринг `comparison.html` из шаблона и данных

## 5. Конфигурация и параметры

Весовые коэффициенты (по умолчанию в ProfileComparer):
| Ключ | Вес | Модуль |
|------|-----|--------|
| name | 0.15 | NameMatcher |
| visual | 0.25 | VisualMatcher |
| friends | 0.20 | FriendsMatcher |
| social_geo | 0.07 | SocialGeoAnalyzer |
| geolocation | 0.08 | GeoMatcher |
| content | 0.15 | ContentMatcher |
| demographics | 0.10 | DemographicsMatcher |

Все параметры настраиваются через `ProfileComparer(custom_weights={...})`.

## 6. Механизмы отказоустойчивости

### Graceful degradation
- Каждый `_analyze_*()` возвращает `{'score': 0.0, 'has_data': False}` при отсутствии данных
- Итоговый `confidence` падает пропорционально количеству отсутствующих данных
- Система работает, но результат менее надёжен

### Чистая изоляция матчеров
- Матчеры не имеют состояния (кроме кэширования)
- Один матчер никогда не вызывает другой напрямую
- Ошибка в одном матче не ломает остальные

## 7. Точки расширения (Extensibility)

Для добавления нового фактора (пример: "общие группы"):
1. Создать `src/matchers/groups_matcher.py` с методом `compare()`
2. Добавить в `ProfileComparer.__init__`: `self.groups_matcher = GroupsMatcher()`
3. Добавить в `DEFAULT_WEIGHTS`: `'groups': 0.05`
4. Добавить метод `_analyze_groups()` с вызовом `groups_matcher.compare()`
5. Вызвать в `compare_profiles()` перед агрегацией
6. Обновить `_generate_detailed_breakdown()`

## 8. Тестируемость

- Матчеры — чистые функции, легко юнит-тестировать
- ProfileComparer — легко мокать матчеры для integration-тестов
- Handlers — тестировать с фикстурами VK API

## 11. Архитектурные решения (Обоснование)

### Почему 7 независимых факторов?
Каждый фактор даёт уникальный сигнал:
- **Имя** — базовый идентификатор
- **Лица** — биометрический (самый сильный, вес 25%)
- **Друзья** — социальный граф (очень надёжен)
- **Социальные геомаркеры** — дополнительный графовый сигнал
- **Контент** — психологическое сходство
- **Демография** — базовые факты
- **Геолокация** — пространственный сигнал

### Почему ProfileComparer централизован?
- Сводит все сигналы в единую формулу
- Позволяет настраивать веса под задачу
- Единая точка контроля интерпретации

### Почему матчеры без состояния?
- Потоковая обработка (один профиль — один вызов)
- Упрощает параллелизм и кэширование
- Тестируемость

## 12. Известные ограничения архитектуры

- Нет асинхронности (синхронные вызовы, медленные face recognition блокируют)
- Нет распределённости (всё в рамках одного процесса)
- Веса фиксированы на этапе вызова (динамическая адаптация не предусмотрена)
- Кэширование только в памяти процесса (нет shared cache между запусками)

## 14. Соглашения об именах

- `src/matchers/` — классы сравнения (`NameMatcher`, `GeoMatcher`, ...)
- `src/handlers/` — обработчики данных
- `src/utils/` — вспомогательные утилиты
- `src/output/` — форматирование результатов
- `compare_*()` — методы сравнения, возвращают dict с `score` и `has_data`
- `analyze_*()` — методы анализа, вызывают compare внутри
- `_*_handler()` — приватные методы ProfileComparer
